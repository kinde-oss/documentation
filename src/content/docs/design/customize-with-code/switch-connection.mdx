Let users switch authentication types within your custom UI page mid-flow.

## Full code example

```jsx title="page.tsx"
export default async function Page(event) {
  const { connections, actions, session } = event.context;
  const available = connections?.available ?? [];
  const switchAction = actions?.switchConnection;
  const psid = session?.pipelineStepId;

  return (
    <div data-kinde-root>
      <h1>Choose how to continue</h1>
      <ul>
        {available.map((connection) => (
          <li key={connection.id}>
            <button
              type="button"
              data-kinde-change-connection-button="true"
              data-kinde-change-connection-id={connection.id}
              data-kinde-change-connection-psid={psid}
              data-kinde-change-connection-action={JSON.stringify(switchAction)}
            >
              Continue with {connection.name}
            </button>
          </li>
        ))}
      </ul>
    </div>
  );
}
```

### Pre-fill an identifier when switching

If the user has already typed an email or username, pass it with `data-kinde-change-connection-login-hint` so the target connection's form is pre-filled:

```jsx
data-kinde-change-connection-login-hint={enteredEmail}
```

For credential connections (for example email + password), Kinde carries the entered credential into the credential screen automatically, so the user does not re-type it.

<Aside>

The connection switch is submitted as a secure `POST`, not a client-side redirect, so the browser back button and the user's session state stay consistent.

</Aside>



When you build your own sign-up and sign-in UI with Kinde page customization, you can let people switch to a different authentication connection from within your page — for example, from an email form to "Continue with Google", or between two enterprise connections. Kinde handles the switch server-side and returns the user to the right next step.

You do not need to add any script. Kinde automatically loads the connection-switch behaviour on custom UI pages. Your job is to render the buttons and tag them with the `data-kinde-change-connection-*` attributes described below.

## How it works

1. Kinde provides the available connections and a switch action in the page `context`.
2. You render a button (or any clickable element) for each connection and add the required data attributes.
3. When the user clicks, Kinde submits a secure `POST` and continues the flow on the chosen connection — redirecting to the credential screen for credential connections, or starting the provider redirect for social and enterprise connections.

## Read the available connections

The page `context` exposes the connections a user can switch to, plus the switch action and the current session identifiers:

```js
export default async function Page(event) {
  const { connections, actions, session } = event.context;

  const available = connections?.available ?? [];
  const switchAction = actions?.switchConnection;
  const psid = session?.pipelineStepId;

  // ...render your UI
}
```

Each entry in `connections.available` looks like this:

```json
{
  "id": "conn_0192...",
  "friendlyId": "google_oauth",
  "name": "Google",
  "connectionType": "social",
  "credentialMethod": null,
  "identityType": "email",
  "provider": "google",
  "logoName": "google"
}
```

| Field | Description |
| --- | --- |
| `id` | Internal connection ID. Use this (or `friendlyId`) as the connection to switch to. |
| `friendlyId` | Human-readable connection identifier. |
| `name` | Display name for the connection. |
| `connectionType` | `social`, `enterprise`, `credential`, or `other`. |
| `credentialMethod` | For credential connections, the method (for example `email:password`, `phone:otp`). `null` otherwise. |
| `identityType` | The identity type, such as `email`, `username`, or `phone`. |
| `provider` | The provider key, such as `google` or `microsoft`. |
| `logoName` | The logo asset name, for rendering a provider icon. |

## Render a switch button

Tag each connection button with the connection-switch data attributes. Kinde reads them on click and submits the switch:

```jsx
<button
  type="button"
  data-kinde-change-connection-button="true"
  data-kinde-change-connection-id={connection.id}
  data-kinde-change-connection-psid={psid}
  data-kinde-change-connection-action={JSON.stringify(switchAction)}
  data-kinde-change-connection-auth-intent="sign_in"
>
  Continue with {connection.name}
</button>
```

### Attributes

| Attribute | Required | Description |
| --- | --- | --- |
| `data-kinde-change-connection-button` | Yes | Set to `"true"` to mark the element as a switch trigger. |
| `data-kinde-change-connection-id` | Yes | The connection to switch to — the `id` or `friendlyId` from `context.connections.available`. |
| `data-kinde-change-connection-psid` | Yes | The current pipeline step ID, from `context.session.pipelineStepId`. |
| `data-kinde-change-connection-action` | Yes | The switch action as a JSON string: `JSON.stringify(context.actions.switchConnection)`. |
| `data-kinde-change-connection-auth-intent` | No | `"sign_in"` (default) or `"sign_up"`. Controls whether the target connection continues as a sign-in or sign-up. |
| `data-kinde-change-connection-login-hint` | No | Pre-fills the identifier (email or username) on the target connection. |
